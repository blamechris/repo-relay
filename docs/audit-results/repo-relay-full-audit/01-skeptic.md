# Skeptic's Audit: repo-relay Full Bug & Quality Audit

**Agent**: Skeptic -- cynical systems engineer; cross-references every claim against actual code
**Overall Rating**: 2.5 / 5
**Date**: 2026-06-09

The code itself is tidier than I expected. The documentation is where the bodies are buried: several load-bearing guarantees in README.md/CLAUDE.md are false in the default (GitHub-hosted runner) configuration — which is exactly the configuration every doc, example, and wizard template generates. Verification: `npm run typecheck` passes; `npm test` shows 210/220 passing (the 10 failures are a local better-sqlite3 NODE_MODULE_VERSION ABI mismatch, not code); committed `dist/` is in sync with `src/` (only a sourcemap byte differs after rebuild).

## Top 5 Findings

### 1. The documented state-persistence recipe does not persist state (Critical)
README.md:251-258 tells hosted-runner users to use `actions/cache` with `key: repo-relay-state-${{ github.repository }}` — a **constant key**. `actions/cache` skips the save step on an exact primary-key hit. So the cache is written once, on the first run, and **never updated again**. Every subsequent run restores the day-one snapshot. Everything built on the DB silently degrades: PR↔message mappings, CI status, `pr_data`. README.md:270 calls this "Persistent with cache" — false. The correct pattern is a per-run key (`...-${{ github.run_id }}`) plus `restore-keys`. CLAUDE.md repeats the same claim ("safe for cache-based persistence"). The WAL-checkpoint-on-close work (state.ts:560-568) was built to support a recipe that doesn't function.

### 2. Following the README's permission list guarantees the bot fails on every event (Critical)
`validatePermissions()` hard-requires six permissions including **Manage Threads** (src/index.ts:70-77) and **throws** if any is missing (index.ts:240-246) — it runs before every event (cli.ts:94). The README's "Required Discord Permissions" table (README.md:215-229) lists only five and **omits Manage Threads**. A user who follows the README's invite-URL instructions gets a bot that connects and then errors on 100% of events. Bonus: README.md:68 instructs enabling "Message Content Intent, Server Members Intent" — privileged intents the code never requests (index.ts:95-100 uses only `Guilds` + `GuildMessages`). Wrong in both directions.

### 3. Timed-out/stale/action-required CI is reported as "✅ Passed" (High)
`mapCiStatus()` (src/handlers/ci.ts:127-137) handles `success`/`failure`/`cancelled` and then `default: return 'success'` with the comment "neutral, skipped treated as success". But GitHub's real conclusion set also includes `timed_out`, `action_required`, `stale`, and `startup_failure` — the payload type at ci.ts:22 simply omits them, so the cast in cli.ts:114 hides it from the compiler. A CI run that times out is rendered as "✅ Passed" in the embed (builders.ts:530-531). For chroxy, an agent-control daemon trusting these notifications, this is actively misleading signal, not just noise.

### 4. Scheduled review polling — a documented feature — is a no-op in the configuration the wizard generates (High)
CLAUDE.md claims scheduled polling "polls all open PRs (from `pr_data` table)... Catches reviews on quiet PRs." Reality chain: `pollOpenPrReviews` → `db.getOpenPrNumbers()` (state.ts:427-432) reads only the local SQLite `pr_data` table, with **no Discord-channel fallback** (unlike message lookup). The setup wizard's template (src/setup/workflow-template.ts:50-53, 92) emits `schedule` + `runs-on: ubuntu-latest` and **no `actions/cache` step**. Fresh runner → empty DB → "No open PRs to poll" (index.ts:449-451) → the feature burns a gateway session every 5 minutes (288/day against Discord's 1000/day session budget) and detects nothing. Even with the README cache recipe, finding #1 means it polls only PRs open at the time of the first-ever cache save. Adjacent bug: workflow-template.ts:23 emits `branches: [$default-branch]` for push events — `$default-branch` is substituted only in GitHub *workflow templates*, not user workflows, so the generated push trigger never fires; the test suite enshrines this bug (workflow-template.test.ts:156).

### 5. "Gracefully recovers" is overstated, and the cascade-fix claim contradicts itself across docs (Medium)
- The channel-search fallback (src/discord/lookup.ts:55-90) recovers the message ID, thread ID, and footer status — but **never `pr_data`**. After state loss, `handleCiEvent` updates the DB then hits `buildEmbedWithStatus` → `getPrData` → null, and both the embed edit *and* the thread post live inside `if (statusData)` (ci.ts:90-109): CI updates are silently dropped for recovered PRs until the next `pull_request` event repopulates `pr_data`. Same structure in comment.ts:96-106 and review.ts:85-95. The fallback also only scans the last 100 channel messages (lookup.ts:26) — older PRs get duplicate embeds/threads via the create-new paths (pr.ts:229-245).
- README.md:287 claims review-reply cascades are "handled automatically since v1. No workflow-level filter needed." The filter (review.ts:52-54, pre-filter.ts:83-87) compares `review.user.login === repository.owner.login` — on an **org-owned repo** the owner is the org, so no human reviewer ever matches and the filter does nothing; on personal repos it filters only the owner. Meanwhile CLAUDE.md ("Onboarding Learnings" #2) still says it "Requires workflow `if` filter. See #13." The two documents flatly contradict each other; the README's version is the false one.

## Recommendations (concrete)

1. **Fix the README cache recipe now**: `key: repo-relay-state-${{ github.repository }}-${{ github.run_id }}` + `restore-keys: repo-relay-state-${{ github.repository }}-`. This single doc bug invalidates most persistence-dependent behavior for hosted runners.
2. Add **Manage Threads** to README.md:219-229 (or drop it from `REQUIRED_PERMISSIONS` if `setArchived(false)` is the only consumer and you're willing to degrade); delete the privileged-intents instruction at README.md:68.
3. In `mapCiStatus`, enumerate `timed_out`/`action_required`/`stale`/`startup_failure` → `failure` and make the unknown-conclusion default `failure` (or at minimum log loudly). Widen the type at ci.ts:22 to match GitHub's actual API.
4. Make `pollOpenPrReviews` fail loudly (or fall back to a GitHub API `GET /pulls?state=open` listing — one extra call) when `pr_data` is empty but the schedule fired; add `actions/cache` to the wizard template whenever `reviewPolling` is enabled; replace `$default-branch` with the resolved branch name (the wizard already talks to the repo) or `main`.
5. Recover `pr_data` in `getExistingPrMessage` — the embed being recovered contains title/URL/branch in its own fields; parse them (or extend the footer metadata to carry them) so CI updates survive state loss. Move the thread-post in ci.ts:99-107 outside the `if (statusData)` block as an interim fix.
6. Replace `errMsg.includes('Unknown Message')` with `error instanceof DiscordAPIError && error.code === 10008` (six call sites); replace `process.exit(1)` in cli.ts's catch with `process.exitCode = 1` so the `finally` checkpoint actually runs.
7. Add a CI step: `npm run build && git diff --exit-code dist/` to make the committed-dist deployment model safe.
8. Pre-filter `issue_comment` bodies against `AGENT_REVIEW_PATTERNS` in `shouldSkipEvent` (the body is in the payload; pre-filter.ts:25-34 already gates on action/PR-ness) — this eliminates the largest class of wasted gateway sessions, which is the root cause the session-limit-retry machinery (#120) treats symptomatically.
9. Resolve the README↔CLAUDE.md contradiction on cascade filtering, and document that the owner-comment filter does nothing for org-owned repos (review.ts:52-54) — chroxy repos under an org will hit this immediately.

## Verdict

repo-relay is a competently written personal tool whose documentation writes checks the default deployment can't cash. The handler code, DB layer, and embed builders are genuinely solid — parameterized SQL, integrity checks, footer-based recovery, 220 real tests. But the three pillars the README sells to hosted-runner users — state persistence via cache, scheduled review polling, and graceful recovery — are respectively broken (constant cache key never re-saves), a silent no-op (empty DB every run), and partial (recovered PRs drop CI updates); and a user following the documented Discord permission list gets a bot that fails on every single event. Add a CI-status mapper that reports timed-out builds as passed, and the "production dependency" bar for chroxy is not met today. The good news: every critical finding is a docs/template/mapping fix, not an architecture problem — a focused day of work on recommendations 1-4 would move this to a 3.5, and the full list to a 4.
